升级到 OpenClaw 2026.3.2 后，很多朋友发现自己的 Agent 突然“变笨”了——不能写代码，不能执行命令，甚至连文件都读不了。

这是因为 OpenClaw 引入了一个重大的 **Breaking Change (破坏性更新)**。

## 什么是 Breaking Change？

在软件开发中，Breaking Change 指的是那些**不向后兼容**的改动。简单说，就是“你以前的代码/配置，在新版本里跑不通了”。

这次 OpenClaw 的更新，最大的改变在于**权限策略的翻转**：

*   **以前 (Legacy)**: 默认信任。Agent 启动即拥有系统权限，想干嘛干嘛。
*   **现在 (v2026.3.2+)**: **默认安全 (Secure by Default)**。

## 发生了什么：Tool Profiles 的默认值变更

OpenClaw 的 **Tool Profiles (工具配置)** 机制其实早就存在，但在此次更新中，**默认配置**从“全能模式”变成了 **`messaging` (通讯员模式)**。

这意味着，除非你显式授权，否则 Agent **默认只能收发消息**。

*   **`minimal`**: 仅保留会话状态，极其受限。
*   **`messaging` (新默认值)**: 仅能聊天。适合客服、通知类 Bot。
*   **`coding`**: 开发者模式。允许读写文件，但限制系统命令。
*   **`full`**: 全功能模式。无限制。

这就像是以前公司发门禁卡，默认给的是“全通卡”；现在新员工入职，默认只给“大堂通行证”。虽然安全了，但对于习惯了“全通卡”的老员工（开发者）来说，确实很不方便。

## 如何解决？

### ❌ 方案一：暴力解锁 (不推荐)

最简单的办法，是直接把配置改回“全通模式”，并且让它在宿主机上直接运行。

```json
// 🚫 危险配置：千万别这么做
{
  "tools": {
    "profile": "full",
    "exec": {
      "host": "gateway",  // 直接在宿主机运行
      "ask": "off"        // 从不询问，直接执行
    }
  }
}
```

**为什么危险？**
这就好比为了方便进出，把公司大门直接拆了。一旦 Agent 出现“幻觉”（比如误执行 `rm -rf /` 或修改了系统关键配置），你的宿主机将没有任何防线，直接崩溃。

### ✅ 方案二：进阶配置 (信任提权 + 自动执行)

我们想要的是：**既要有权限的便利（自动化），又要有“防爆门”的安全（沙箱）。**

这里推荐一个 **“默认沙箱 + 信任提权 + 自动执行”** 的配置方案。

#### 核心逻辑

1.  **默认安全 (Default Sandbox)**: 所有普通命令（如 `ls`, `git`, `npm`）依然在 **Docker 沙箱** 中运行。即使 Agent 发疯删文件，也只是删了容器里的临时文件，宿主机毫发无损。
2.  **按需提权 (Elevated Privileges)**: 当 Agent 需要执行宿主机敏感操作（如 `openclaw` 管理命令）时，它会发起提权请求。
3.  **自动批准 (Auto-Approve)**: 只要是你（信任用户）发起的会话，系统自动批准提权，不再弹窗打断心流。

#### 配置示例

修改 `~/.openclaw/openclaw.json`：

```json
{
  "tools": {
    "profile": "full",   // 1. 恢复全功能 Profile
    "exec": {
      "host": "sandbox",   // 2. 默认依然是沙箱，防止误伤
      "security": "full",  // 3. 信任模式，不做白名单检查
      "ask": "off"         // 4. 关闭审批弹窗，彻底自动化
    },
    "elevated": {
      "enabled": true,
      "allowFrom": {
        // 5. 只有这里列出的用户才能触发上述的“自动提权”
        "feishu": ["ou_YOUR_FEISHU_ID"]
      }
    }
  }
}
```

> **💡 小技巧**
>
> 修改完配置后，你可以使用 v2026.3.2 新增的命令来验证配置是否合法：
> ```bash
> openclaw config validate --json
> ```
> 这样可以避免重启 Gateway 后才发现配置错误。

### 为什么这个方案更好？

让我们对比一下直接开放 Full 权限和这个中间方案的区别：

| 场景 | 暴力解锁 (`host: gateway`) | 进阶方案 (`host: sandbox` + 提权) |
| :--- | :--- | :--- |
| **智能体运行 `rm -rf /`** | 💥 **直接执行，系统崩溃** | 🛡️ **在沙箱内执行，宿主机安全** |
| **智能体运行 `openclaw ...`** | ✅ 直接执行 | ✅ **自动提权并执行 (无感)** |
| **安全性** | 🔴 极低 (裸奔) | 🟡 中等 (仅信任你的提权操作) |
| **体验** | 🚀 顺滑 | 🚀 顺滑 |

## 结语

OpenClaw 2026.3.2 的这次 Breaking Change，本质上是从“玩具”走向“生产力工具”的必经之路。

**安全不再是去消除不确定性，而是去包容不确定性。**

通过 **Tool Profiles (逻辑约束)** + **Sandboxing (物理隔离)** + **Trust Elevation (信任机制)**，我们终于可以在不牺牲安全的前提下，享受 AI Agent 带来的极致自动化体验。
